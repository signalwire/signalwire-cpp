# PORT_ADDITIONS.md -- C++-specific additions not present in the Python reference

This file is checked by `scripts/enumerate_surface.py` +
`porting-sdk/scripts/diff_port_surface.py` on every PR. Every C++ symbol
that has no Python-reference equivalent MUST appear below with a rationale,
or the surface audit fails. Reviewers use this to spot accidental drift
from the reference API. See `porting-sdk/CHECKLIST_TEMPLATE.md` section 13
"Symbol-level surface parity".

**Line format** -- one `<fully.qualified.symbol>: <reason>` per line.

## Rationale vocabulary

- `python_mixin_collapsed`: the Python reference fans these methods out
  across mixin classes; the C++ port collapses them onto the concrete
  class (AgentBase in particular). `scripts/enumerate_surface.py`
  projects each method back to its mixin module so the diff still
  matches, but the flat copy on the concrete class is also a valid
  symbol -- this entry documents that duplicate view.
- `cpp_only`: no Python-reference equivalent at all; intentional C++
  extension (e.g., typed enum `PhoneCallHandler`, static helper
  `timing_safe_compare`, per-namespace struct types on `RestClient`).
- `cpp_typed_*`: the C++ API exposes a typed accessor/helper for
  ergonomics where Python uses dynamic dispatch or dict-keyed lookup.
  Not load-bearing for parity -- Python users reach the same state via
  attribute reads.
- `cpp_naming`: method under a C++-idiomatic name where the Python
  reference uses a different name (e.g., `render_swml` vs
  `render_document`, `to_json` vs `to_dict`). The Python-named variant
  is tracked as a `not_yet_implemented` alias in PORT_OMISSIONS.md.
- `cpp_structural_type` / `cpp_namespace_type` / `cpp_only_pod`: C++
  POD struct or typed namespace container used internally; Python uses
  untyped dicts/attribute access.
- `cpp_unified_action` / `cpp_unified_event`: C++ uses a single
  generalized RELAY Action/event type where Python ships per-verb
  subclasses; the subclasses are tracked in PORT_OMISSIONS.md as
  `not_yet_implemented`.
